# Sunny Tea House AI奶茶评价生成系统 完整需求分析文档
## 一、文档概述
### 1.1 项目背景
本项目基于虚构奶茶店「Sunny Tea House」（地址位于美国圣何塞 San Jose）开发，打造适配移动端Chrome浏览器的H5评价互动平台。平台覆盖普通顾客、店铺商家、系统管理员三类角色，核心能力为AI辅助生成多平台风格的奶茶店评价，同时支持商铺浏览、评价互动、商家运营、后台管理等完整业务闭环。系统采用前后端分离架构，以单Docker镜像形式部署，轻量化交付、可直接在CentOS 7环境运行。

### 1.2 技术栈明细
| 分层 | 技术选型 | 补充说明 |
|------|----------|----------|
| 前端 | Vue 3 + Vite | 纯移动端H5，适配手机Chrome浏览器，触屏交互优化 |
| 后端 | FastAPI（全异步 async/await） | 异步接口、异步大模型调用、异步数据库操作，直接托管前端静态资源 |
| 数据库 | MySQL 8.0 | 业务数据持久化存储 |
| ORM框架 | SQLAlchemy 2.0 异步ORM | 统一数据库操作，规避原生SQL编写 |
| 中间件 | Redis 6.x | 用户会话缓存、商铺数据缓存、接口限流、好评率统计缓存 |
| 部署方式 | Docker 单镜像打包 | 无Nginx依赖，单容器启动完整服务 |
| 运行环境 | CentOS 7 | 兼容Linux内核，支持标准Docker命令 |
| AI能力 | 第三方大模型API + Prompt工程 | 密钥预留配置位，支持轻量化微调方案 |
| 资源存储 | 服务器本地目录 | 商铺图片、商品图片均存储在本地文件系统 |

### 1.3 运行载体
- 核心适配：手机端Chrome浏览器
- 兼容范围：主流安卓移动端浏览器，全页面触屏交互优化

### 1.4 核心约定
1. **角色与账号规则**
    - 系统内置超级管理员账号：`admin` / `123456`，系统初始化时自动写入数据库，密码bcrypt哈希加密，禁止前端注册入口；
    - 普通用户、商家用户均支持前端自主注册；
    - 系统初始无预置商铺、商品、普通用户数据，全部由使用者自行添加。
2. **AI能力约定**
    - 大模型API密钥不硬编码，存储于系统配置表，支持手动修改配置；
    - AI评价生成强制绑定选中商铺信息，仅生成对应店铺的定制化评价。
3. **权限约定**
    - 游客（未登录）：仅可浏览商铺列表、商铺详情、公开评价列表，不可发布评价、收藏、点赞、回复；
    - 普通用户：可发布评价、AI生成评价、收藏商铺、浏览历史、评价互动（点赞/回复）；
    - 商家用户：可管理自身店铺信息、添加商品、回复用户评价；
    - 管理员：全平台数据管控、商铺推荐设置、系统配置管理。
4. **功能取舍约定**
    - 保留原始Demo的「双平台风格生成」「一键复制跳转平台」「消费感受标签选择」能力；
    - 取消企业微信群机器人Webhook推送功能。

## 二、功能需求详细分析
### 2.1 普通用户端功能（顾客角色）
#### 2.1.1 账号模块
1. **用户注册**
    - 输入账号、密码完成注册，账号全局唯一，重复账号前端提示；
    - 密码后端通过bcrypt加盐哈希加密后存入数据库，不存储明文；
    - 注册成功自动登录，返回有效Token。
2. **用户登录**
    - 输入账号密码提交校验，后端比对数据库哈希密码；
    - 校验通过生成随机UUID，经哈希加密后作为Redis会话Key；
    - 用户基础信息存入Redis，前端仅获取哈希后的Token Key，不存储敏感数据；
    - Token有效期可配置，默认24小时。
3. **退出登录**
    - 清除Redis中对应用户会话，前端Token失效。

#### 2.1.2 商铺浏览模块
1. **商铺列表页**
    - 展示所有上架状态的商铺，支持两种排序：手动推荐优先、好评率从高到低排序；
    - 展示商铺门面图、名称、人均消费、好评率、推荐标签；
    - 系统自动计算每个商铺的好评率，实时更新。
2. **商铺详情页**
    - 展示商铺门面图、详细位置、人均消费、商铺简介、好评率；
    - 展示该商铺下的全部商品列表（名称、价格、图片、描述）；
    - 展示该商铺所有用户评价，按发布时间倒序排列；
    - 登录状态下自动记录用户浏览历史。
3. **浏览历史**
    - 登录用户可查看自己访问过的商铺记录，按浏览时间倒序；
    - 支持单条删除、一键清空历史记录。
4. **商铺收藏**
    - 登录用户可收藏/取消收藏商铺；
    - 「我的收藏」页面查看所有收藏的商铺，支持快速跳转详情。

#### 2.1.3 AI评价生成核心模块
1. **评价维度选择**
    - 评价等级：必选，分为「推荐」「一般」「差评」三类；
    - 消费感受标签：必选1~2个，包含「服务好」「出餐快」「环境干净」「饮品颜值高」「性价比高」「口味正宗」等；
    - 目标发布平台：必选，分为「Google」「小红书」两类。
2. **AI智能生成**
    - 基于当前商铺信息、评价等级、选中标签、目标平台，调用大模型生成定制化评价；
    - Google风格：英文输出，符合北美本地消费者客观评论口吻；
    - 小红书风格：中文输出，典型种草笔记风格，包含Emoji、分段排版、呼吸感格式；
    - 生成过程展示加载状态，禁止重复提交。
3. **评价编辑与提交**
    - AI生成的文案支持用户二次编辑修改；
    - 编辑完成后可提交至站内评价，绑定用户ID、商铺ID、评价等级、标签、目标平台；
    - 提交成功后刷新商铺评价列表，同步更新商铺好评率。
4. **一键复制并跳转**
    - 点击按钮将当前评价文案复制到手机系统剪贴板；
    - 根据所选目标平台，尝试唤起对应App，唤起失败自动跳转平台网页发布入口；
    - 无需实现内容自动预填，仅完成跳转唤起。

#### 2.1.4 评价互动模块
1. **评价点赞**
    - 登录用户可对任意评价点赞，取消点赞；
    - 单用户对单评价仅可点赞一次，自动去重。
2. **评价回复**
    - 登录用户可对评价发布回复评论；
    - 支持多层级回复展示，按时间正序排列；
    - 可查看回复发布者、发布时间、回复内容。

#### 2.1.5 个人中心
- 个人账号信息展示；
- 我的收藏、浏览历史、我的评价入口；
- 退出登录按钮。

### 2.2 商家端功能（店铺运营角色）
#### 2.2.1 账号与店铺绑定
1. 商家账号支持前端自主注册，注册后角色为「商家」；
2. 商家登录后可创建/编辑自身绑定的商铺信息，一个商家账号对应一个商铺。

#### 2.2.2 商铺信息管理
- 编辑商铺基础信息：商铺名称、门面图片、详细地址、人均消费、商铺简介；
- 图片上传至服务器本地目录，后端统一管理静态资源路径；
- 商铺上架/下架操作（下架后用户端不可见）。

#### 2.2.3 商品信息管理
- 为自身商铺新增商品：商品名称、价格、商品图片、商品描述、排序权重；
- 支持编辑、删除、上下架商品；
- 商品按排序权重正向展示。

#### 2.2.4 评价管理与回复
- 查看自身商铺下的所有用户评价，支持按评价等级筛选；
- 针对用户评价发布商家官方回复，回复带有「商家」标识；
- 查看评价点赞数、回复数统计。

### 2.3 管理员端功能（系统管理员）
#### 2.3.1 用户管理
- 分页查看所有注册用户（普通用户+商家用户），支持按账号、角色搜索；
- 查看用户基础信息、发布评价数、收藏数；
- 支持禁用/启用用户账号，禁用后用户无法登录；
- 支持重置用户密码。

#### 2.3.2 评价管理
- 查看全平台所有用户评价，支持按商铺、评价等级、发布时间、目标平台筛选；
- 查看评价详情、对应发布用户、所属商铺、点赞回复数据；
- 支持删除违规评价，删除后同步更新商铺好评率统计。

#### 2.3.3 商铺管理
1. **商铺全量管控**
    - 查看所有商铺信息，支持下架违规商铺；
    - 手动设置/取消「推荐商铺」标签，推荐商铺在列表顶部优先展示。
2. **自动好评率统计**
    - 系统自动计算每个商铺的好评率 = 「推荐」等级评价数 / 总有效评价数；
    - 支持按好评率自动排序，好评率数据缓存至Redis提升查询性能。

#### 2.3.4 系统配置管理
- 查看/修改大模型API密钥、模型名称、请求超时时间；
- 查看/修改Token有效期、商铺缓存过期时间、AI每日调用限流阈值等系统参数；
- 所有配置修改后实时生效。

## 三、非功能需求
### 3.1 安全需求
1. **密码安全**：所有用户密码采用bcrypt加盐哈希加密，数据库无明文密码存储；
2. **会话安全**：Token为哈希加密后的随机UUID，用户信息仅存Redis，前端仅持有Key，避免敏感信息泄露；
3. **权限隔离**：三类角色接口权限严格隔离，后端通过角色鉴权统一拦截，普通用户不可访问商家、管理员接口；
4. **接口权限**：游客仅开放商铺浏览、评价查看类读接口，所有写操作强制登录；
5. **密钥安全**：大模型密钥存储于数据库配置表，代码无硬编码，前端不可获取。

### 3.2 性能需求
1. 全异步架构：FastAPI异步路由、异步SQLAlchemy、异步Redis、异步大模型调用，阻塞操作不占用主线程；
2. 缓存加速：商铺基础信息、好评率、商品列表全量缓存Redis，核心查询接口优先走缓存，减少数据库压力；
3. 响应指标：商铺列表、详情类接口响应时间≤200ms；AI生成评价响应≤5s（取决于大模型接口速度）。

### 3.3 兼容性需求
1. 前端适配手机端Chrome浏览器，兼容安卓主流内置浏览器，自适应不同屏幕宽度；
2. Docker镜像兼容CentOS 7系统内核，支持标准Docker命令启动。

### 3.4 可维护性需求
1. 系统参数全配置化，修改大模型密钥、缓存时效、限流阈值无需改动代码；
2. 业务分层清晰：路由层、业务层、数据层、第三方调用层解耦；
3. 缓存自动更新：商铺、评价、商品数据变动时主动清除对应缓存，保证数据一致性。

## 四、数据库表结构设计（SQLAlchemy ORM）
### 4.1 表总览
| 表名 | 表说明 |
|------|--------|
| `sys_user` | 用户账号表（普通用户/商家/管理员） |
| `shop` | 商铺基础信息表 |
| `shop_product` | 商铺商品表 |
| `user_evaluation` | 用户评价主表 |
| `evaluation_reply` | 评价回复表（用户+商家回复） |
| `evaluation_like` | 评价点赞表 |
| `user_collection` | 用户收藏关联表 |
| `user_browse_history` | 用户浏览历史表 |
| `system_config` | 系统全局配置表 |

### 4.2 分表详细结构
#### 表1：sys_user 用户账号表
| 字段名 | 数据类型 | 约束 | 字段说明 |
|--------|----------|------|----------|
| `id` | BIGINT | 主键、自增 | 用户唯一ID |
| `username` | VARCHAR(50) | 唯一、非空 | 登录账号 |
| `password_hash` | VARCHAR(255) | 非空 | bcrypt哈希加密后的密码 |
| `role` | ENUM('user','shop','admin') | 非空、默认'user' | 用户角色：普通用户/商家/管理员 |
| `is_active` | TINYINT(1) | 非空、默认1 | 账号状态：1启用 0禁用 |
| `create_time` | DATETIME | 非空、默认当前时间 | 账号创建时间 |
| `update_time` | DATETIME | 更新时自动刷新 | 最后修改时间 |

> 初始化约定：系统启动时自动检查是否存在admin账号，不存在则插入内置账号（admin/123456），密码自动哈希加密；无其他预置账号。

#### 表2：shop 商铺基础信息表
| 字段名 | 数据类型 | 约束 | 字段说明 |
|--------|----------|------|----------|
| `id` | BIGINT | 主键、自增 | 商铺唯一ID |
| `owner_id` | BIGINT | 外键关联sys_user.id、非空 | 所属商家用户ID |
| `shop_name` | VARCHAR(100) | 非空 | 商铺名称 |
| `cover_image` | VARCHAR(500) | 非空 | 商铺门面图片本地路径 |
| `address` | VARCHAR(200) | 非空 | 商铺位置地址 |
| `avg_cost` | DECIMAL(10,2) | 非空 | 人均消费金额 |
| `description` | TEXT | 空 | 商铺简介 |
| `good_rate` | DECIMAL(5,2) | 非空、默认0.00 | 商铺好评率，自动计算更新 |
| `is_recommended` | TINYINT(1) | 非空、默认0 | 管理员手动推荐：1是 0否 |
| `is_active` | TINYINT(1) | 非空、默认1 | 是否上架：1上架 0下架 |
| `create_time` | DATETIME | 非空、默认当前时间 | 创建时间 |
| `update_time` | DATETIME | 更新时自动刷新 | 最后修改时间 |

#### 表3：shop_product 商铺商品表
| 字段名 | 数据类型 | 约束 | 字段说明 |
|--------|----------|------|----------|

| `product_name` | VARCHAR(100) | 非空 | 商品名称 |

| `create_time` | DATETIME | 非空、默认当前时间 | 创建时间 |

#### 表4：user_evaluation 用户评价主表
| 字段名 | 数据类型 | 约束 | 字段说明 |
|--------|----------|------|----------|

|--------|----------|------|----------|

## 五、Redis缓存设计
严格遵循「Session仅存储Redis Key，用户数据存Redis」的要求：

### 5.1 用户会话缓存
- **Key格式**：`session:{hash_uuid}`（哈希加密后的UUID作为Key）
- **Value内容**：用户基础信息JSON（user_id、username、role）
- **过期时间**：与Token有效期一致，默认24小时
- **交互规则**：前端请求携带Token（即hash_uuid），后端从Redis读取用户信息；Session层仅存储Key，不持久化用户敏感数据。

### 5.2 商铺数据缓存
1. **商铺列表缓存**
    - Key：`shop:list:active`
    - Value：所有上架商铺的基础信息（含好评率、推荐标记）JSON
    - 过期时间：默认1小时
2. **商铺详情缓存**
    - Key：`shop:detail:{shop_id}`
    - Value：商铺详情+对应商品列表JSON
    - 过期时间：默认1小时
3. **商铺好评率缓存**
    - Key：`shop:good_rate:{shop_id}`
    - Value：好评率数值
    - 过期时间：默认30分钟，评价新增/删除时主动更新

### 5.3 限流缓存
- **Key格式**：`limit:ai:{user_id}:{date}`
- **Value**：当日AI生成调用次数
- **过期时间**：自然日过期，用于控制单用户每日大模型用量。

## 六、AI评价模块设计
### 6.1 核心规则
1. **强绑定商铺**：AI评价生成必须注入当前商铺名称、品类、人均消费等信息，仅生成对应店铺的定制化评价，禁止生成通用无差别内容；
2. **多维度控制**：通过「评价等级 + 消费标签 + 目标平台」三个维度控制生成风格，确保输出符合预期；
3. **密钥可配置**：大模型API密钥存储于系统配置表，无硬编码，支持后台手动修改。

### 6.2 生成流程
1. 用户在商铺详情页选择评价等级、1~2个消费标签、目标发布平台，点击「AI生成评价」；
2. 后端校验用户登录状态、Redis限流校验；
3. 读取商铺基础信息，结合等级、标签、平台参数，组装定制化Prompt；
4. 异步调用大模型API，生成对应风格的评价文案；
5. 返回文案至前端，支持用户二次编辑，可提交站内评价或一键复制跳转外部平台。

### 6.3 Prompt轻量化方案（满足需求）
采用**Prompt工程 + 店铺信息注入**的轻量化方案，无需额外训练算力，适配Demo场景：
- 系统预设两套平台基础Prompt模板，分别对应Google英文客观评论、小红书中文种草风格；
- 生成时动态注入商铺名称、人均消费、评价等级、选中标签，约束模型输出范围；
- 通过角色设定、输出格式约束、负面示例，保证生成内容符合平台风格与评价等级。

### 6.4 模型微调补充方案（进阶可选）
#### 6.4.1 微调目标
针对奶茶店评价场景优化输出稳定性，减少通用化内容，提升风格匹配度。
#### 6.4.2 训练数据集建议
- 数据规模：500~1000条标注样本，分为Google英文评价、小红书中文评价两类；
- 数据字段：商铺信息、评价等级、消费标签、对应真实评价文案；
- 数据来源：公开平台奶茶店评价爬取 + 人工标注清洗；
- 标注规则：按推荐/一般/差评分级，标注对应消费标签与平台风格。
#### 6.4.3 推荐训练环境
- 框架：LoRA轻量化微调，基于开源大模型（如Llama 3、Qwen2）；
- 硬件：单张16G显存GPU（如RTX 3090/4060Ti）即可完成微调；
- 部署：微调后模型可通过API形式对接，或继续使用第三方大模型+Prompt工程的主方案。

## 七、部署方案设计
### 7.1 部署原则
- 无Nginx依赖：FastAPI直接托管Vue打包后的静态资源，同时提供API接口，单镜像启动完整服务；
- 镜像兼容CentOS 7系统内核，支持标准Docker命令；
- MySQL、Redis采用外部部署，通过环境变量配置连接地址。

### 7.2 Docker镜像构建方案
采用**多阶段构建**，实现镜像轻量化：
1. **第一阶段：前端构建**
    - 基础镜像：`node:18-slim`
    - 执行：安装依赖、打包Vue项目，生成dist静态资源目录。
2. **第二阶段：后端运行**
    - 基础镜像：`python:3.10-slim`（兼容CentOS 7内核）
    - 执行：安装Python依赖、复制前端dist目录、配置静态资源路由、启动FastAPI服务。
3. 镜像特点：单镜像包含前后端全部能力，仅暴露单个端口。

### 7.3 CentOS 7运行方式
```bash
# 加载镜像
docker load -i ai-tea-evaluation.tar

# 启动容器
docker run -d \
  -p 8000:8000 \
  -v /opt/tea-shop/upload:/app/upload \
  -e MYSQL_URL="mysql+aiomysql://用户名:密码@数据库IP:3306/数据库名" \
  -e REDIS_URL="redis://RedisIP:6379/0" \
  --name ai-tea-evaluation \
  ai-tea-evaluation:latest
```
- 本地图片目录通过数据卷挂载，容器销毁不丢失图片资源。

### 7.4 初始化逻辑
- 容器启动时自动执行数据库表结构创建；
- 自动检查admin账号，不存在则自动创建内置管理员账号；
- 自动初始化系统配置表，大模型密钥默认留空，需手动配置后启用AI功能。

## 八、业务核心流程图
### 8.1 AI评价生成主流程
1. 用户登录 → 进入商铺详情 → 选择评价等级+消费标签+目标平台
2. 提交生成请求 → 后端限流校验 → 读取商铺信息组装Prompt
3. 调用大模型生成文案 → 返回前端 → 用户编辑
4. 分支1：提交站内评价 → 写入数据库 → 更新商铺好评率
5. 分支2：一键复制跳转 → 复制文案到剪贴板 → 唤起/跳转目标平台

### 8.2 好评率自动计算流程
1. 新增/删除评价 → 触发异步统计任务
2. 统计商铺总评价数、推荐等级评价数
3. 计算好评率 → 更新shop表good_rate字段 → 清除Redis商铺缓存
4. 用户端列表自动按最新好评率排序
